ここに書かれた指示は、Cursor / Codex / Claude など、このリポジトリで動くすべての AI コーディングエージェントに対する規約です。

## 0. 最重要ポリシー: 「作りつつ学べる」を最優先する

オーナーは Astro を学ぶためにこのブログを作っています。
**動くコードを最短で出すことよりも、オーナーが Astro を理解した状態で前進できることを優先してください。**

エージェントは以下を厳守すること:

1. **いきなり完成形を書かない。**
   - 大きな機能は一発で実装せず、最小実装 → 動作確認 → 拡張、の段階で進める。
   - 1 つの変更で複数ファイルにまたがる大改修をするときは、**着手前に「何を / なぜ / どの順で」変えるか日本語で説明**してから手を動かす。

2. **Astro 固有の概念が出てきたら必ず短く解説する。**
   - 例: `.astro` のフロントマター、Islands Architecture、`client:*` ディレクティブ、Content Collections、`getStaticPaths`、`Astro.props`、レイアウトのスロット、ビューア・トランジションなど。
   - 解説は「3〜6 行 + 公式ドキュメントへのリンク」を目安にコンパクトに。長文の教科書は不要。
   - 公式ドキュメントは <https://docs.astro.build/> を一次情報として参照する。

3. **コードに対して「なぜそう書いたか」を添える。**
   - 代替案がある場合は 1〜2 行で他の選択肢にも触れる(例: 「SSR にする手もあるが、このブログは静的配信で十分なので SSG を選ぶ」)。
   - ただし冗長な解説コメントをコード中に書き散らさないこと。説明はチャット側、コードは簡潔に。

4. **オーナーが書ける部分はオーナーに書かせる余地を残す。**
   - 学習価値が高い箇所(初めて触る API、初めて使う概念)は、骨組みだけ提示して「ここはこう書いてみてください」と促す選択肢を提案する。
   - 逆にボイラープレート(設定ファイル、型定義、既知のバグ修正など)はそのまま書いて構わない。

5. **「魔法」を残さない。**
   - 自動生成されたファイルやテンプレートをそのまま受け入れず、各ファイルの役割を 1 行で説明できる状態にする。
   - わからない記法・props・ディレクティブが混入したら、そのまま流さず短く解説する。

## 1. プロジェクト概要

- 種別: 個人ブログ(静的サイト想定)
- 目的: ブログとして使えるものを作りながら Astro を学ぶ
- 公開先: GitHub Pages

## 2. 技術スタック

| 区分 | 採用 | 補足 |
| --- | --- | --- |
| フレームワーク | **Astro 6** | `astro.config.mjs` の `defineConfig` で設定 |
| UI アイランド | **React**(`@astrojs/react`) | インタラクティブ部分のみ使う想定。基本は `.astro` で書く |
| パッケージマネージャ | **pnpm** | `pnpm-lock.yaml` をコミット |
| Node | **>= 22.12.0** | `package.json` の `engines` で固定 |
| 言語 | TypeScript | `tsconfig.json` ベース |

新しい依存を追加するときは、**まず Astro 標準機能で実現できないか確認**してから提案すること。
(例: ルーティング・画像・フィードなどは Astro に組み込み機能や公式インテグレーションがある)

## 3. ディレクトリ構造とその意味

```
src/
├─ assets/        # Astro の画像最適化対象。`import` して <Image /> で使う
├─ components/    # 再利用 UI。.astro が基本、必要なら .tsx(React) も可
├─ layouts/       # ページの外枠(<html> から </html> までを含む)
└─ pages/         # ファイルベースルーティング。ここに置いたものが URL になる
```

ファイルを追加・移動するときは、**それが Astro のどの機構に乗るのか**を一言添える(例: 「`src/content/` を作って Content Collections に乗せる」)。

## 4. コーディング規約

- `.astro` ファイル優先。インタラクションが必要な部分だけ React コンポーネント(`.tsx`)化する。
- React を使うときは `client:load` / `client:visible` / `client:idle` の **使い分けに必ず触れる**(なぜそのディレクティブを選んだか)。
- スタイルは原則 `.astro` のスコープ付き `<style>` を使う。グローバル CSS を増やす前に相談する。
- 型は基本 strict。`any` を入れる場合は理由をコメントで残す(コード上で正当化が必要な唯一のケース)。
- コミットメッセージは Conventional Commits 風(`feat:` / `fix:` / `docs:` / `chore:` / `refactor:`)。

## 5. よく使うコマンド

| コマンド | 用途 |
| --- | --- |
| `pnpm install` | 依存をインストール |
| `pnpm dev` | 開発サーバ(`http://localhost:4321`) |
| `pnpm build` | `./dist/` に本番ビルド |
| `pnpm preview` | ビルド結果のローカル確認 |
| `pnpm astro check` | 型・コンテンツの静的検査 |
| `pnpm astro add <integration>` | インテグレーション追加(MDX, Sitemap など) |

エージェントが勝手に新しいスクリプトを `package.json` に増やさないこと。必要なら提案ベースで。

## 6. 学習トピックの推奨ロードマップ

着手の参考に。順番は厳守ではなく、ブログ機能を作る過程で自然に踏むこと。

1. ページとレイアウト(`src/pages`, `src/layouts`, スロット)
2. コンポーネントと Props(`Astro.props`, スロット名付き)
3. Markdown / MDX で記事を書く
4. **Content Collections**(`src/content/` + `defineCollection`) ← ブログの肝
5. 動的ルート(`[slug].astro` と `getStaticPaths`)
6. 画像最適化(`astro:assets` の `<Image />` / `<Picture />`)
7. RSS / Sitemap / OGP
8. Islands と `client:*` ディレクティブ(必要になったら React を投入)
9. デプロイ(アダプタの選定 → CI)

新しい機能を追加するときは「これはロードマップのどこに当たるか」を 1 行触れると、学習の地図が崩れない。

## 7. エージェントへの最終チェックリスト

PR / 変更をまとめる前に自己点検:

- [ ] 変更の意図と Astro 上の位置づけを説明したか
- [ ] 新しく登場した Astro 概念に短い解説を添えたか
- [ ] オーナーが手で書く余地のある部分を奪っていないか
- [ ] 不要な依存・抽象化を増やしていないか
- [ ] `pnpm build` と `pnpm astro check` が通る状態か(該当する場合)
